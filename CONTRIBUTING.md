# Contributing to Riverbraid v1.0
Thank you for your interest in contributing! All contributions must follow these non-negotiable rules to ensure consistency, clarity, and maintainability of the codebase.

1. Code Style
	•	Follow PEP 8 for Python code.
	•	Use snake_case for functions and variables, PascalCase for classes.
	•	Add docstrings for all functions, classes, and modules using the Google style.
	•	Maintain clear, self-explanatory naming; avoid abbreviations unless they are widely understood.
	•	No trailing whitespace; always run a linter before committing.

2. Git Branches
	•	Work on feature branches off main.
	•	Branch naming convention: feature/<descriptive-name> or bugfix/<descriptive-name>.
	•	Keep commits small, atomic, and descriptive.

3. Pull Requests (PRs)
	•	PRs must be submitted against the main branch.
	•	Include a clear title and description of the changes.
	•	Link any relevant issues with Closes #<issue_number> if applicable.
	•	PRs must pass all tests and linting checks before merging.
	•	All code must be reviewed and approved by the repository owner (Michael) before merging.

4. Testing
	•	Include tests for all new functionality.
	•	Tests must pass on all supported Python versions.
	•	Use pytest and follow the existing test organization.

5. Documentation
	•	Update relevant documentation for new features or changes.
	•	Ensure that examples are reproducible and code snippets are accurate.

⸻

Strict Rules Summary:
	1.	Adhere to style guides.
	2.	Work on separate branches.
	3.	No merge without review.
	4.	All changes must be tested and documented.

Non-compliant PRs will be rejected immediately.
